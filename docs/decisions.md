# Architecture Decisions

## Auth: Clerk (not NextAuth)
- `User.id = Clerk userId` — eliminates a mapping join on every request
- Built-in webhooks for user creation event (sync to our DB)
- `auth()` in server components/routes is zero-boilerplate
- Free tier is generous for MVP

## Database: Neon + Prisma (not Supabase)
- Simpler integration — just a Postgres URL, no Supabase client SDK layer
- Prisma `Json` fields for scope arrays (deliverables, exclusions, etc.) — type-safe, no manual JSON.parse
- Enums for `ProjectType` and `SubscriptionStatus` — compile-time safety, fewer corrupt records
- Neon's serverless driver is Vercel-optimized

## AI: Anthropic Claude Haiku
- Used for **polishing only** — structured scope from form → prose
- Does NOT invent scope (user provides all items via Scope Builder)
- Haiku: fast (<3s response), cheap (~$0.003/proposal), sufficient quality for template prose
- Upgrade path: swap model ID to Sonnet 3.5 if quality issues reported

## Subscription status: source of truth
`customer.subscription.created/updated` events are the source of truth for `Subscription.status` and `currentPeriodEnd`.
`invoice.payment_failed` is supplemental (used for logging/alerting only, not for status updates).
Rationale: subscription events carry full state; invoice events can arrive out of order.

## Webhook architecture: handleStripeEvent is a pure function
Route handler (`/api/webhooks/stripe/route.ts`) only:
1. Reads raw body
2. Verifies Stripe signature
3. Calls `handleStripeEvent(event)` from `src/lib/stripeWebhook.ts`

`handleStripeEvent` is a testable unit — Jest tests mock Prisma directly without needing Next.js route setup.

## DOCX (not PDF)
- Clients expect editable files — DOCX is the professional standard
- `docx` npm package is pure JS (no system deps, Vercel-compatible)
- PDF export is a v2 feature

## No email in MVP (Resend is optional v2)
- MVP value = DOCX download. Email delivery adds complexity without changing core value.
- If added: send a link to the proposal page, not an attachment

## Watermark implementation
Free tier: single-section DOCX with footer text "Generated by ScopeShield · Not legal advice · scopeshield.io"
Appears on most pages in standard Word/Google Docs rendering.
No guarantee of "every page" across all edge cases (multi-section docs, special print settings).
This is acceptable for MVP — not a support trap.

## Entitlements: enforced in TWO places
1. `/api/generate` — free monthly limit (3/month) or active Pro subscription required
2. `/api/download/[id]` — ownership verified + Pro = clean DOCX, Free = watermarked
Server-side only. No client-side-only gating.

## Rate limiting: in-memory Map
Simple `Map<userId, timestamp>` — 1 req/10s per user.
Good enough for MVP; replace with Redis or Upstash if scaling.
